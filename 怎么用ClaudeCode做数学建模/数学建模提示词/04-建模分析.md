进入建模分析阶段。使用 math-modeling skill 的建模手角色。

⚠️ 主代理没有多模态能力，不能直接读取或识别任何图片。遇到题目中的示意图、图表、手写公式等图像内容，必须派发子代理并指定 model: "opus" 处理。
⚠️ 必须严格按以下顺序逐步执行，不得跳步、合并或自行调整顺序。每完成一步后必须暂停等用户确认，确认后才能进入下一步。

前置读取：
- 读取前置产出（如有读取不到的或是缺失的文件，一定要及时反馈！）：
  - 设计文档（docs/superpowers/specs/ 下的 .md 文件，brainstorming 阶段产出）
  - 计划文档（docs/superpowers/plans/ 下的 .md 文件，writing-plans 阶段产出）
- 读取建模手的角色文档：math-modeling skill 目录下的 references/roles/建模手/SKILL.md
  - 注意：SKILL.md 中的路径是相对于 math-modeling skill 目录的，以下步骤中的路径是完整路径

步骤 0/5：附件检查
→ 检查项目目录下是否有附件数据文件夹（如附件/、data/ 等）
→ 如果有 Excel/CSV 数据文件，使用 xlsx skill 读取，了解数据结构、列名、规模、缺失值情况
→ 如果题目是 PDF，使用 pdf skill 提取文字和表格内容
→ 题目中如有示意图、图表等图像内容，必须派发子代理并指定 model: "opus" 识别，prompt 中要求：先判断图片类型（系统拓扑图/流程图/几何示意图/数据图表/公式约束/参数表/地图空间/图例编码）、系统性观察各区域不猜测、提取所有文字数字公式、描述拓扑连接关系、提取图例颜色编码、提取参数值单位约束、有子图则逐个识别，输出按"图片类型→文字→拓扑→参数→图例"分类整理
→ 确认：数据是否充足、是否有缺失值需要处理、数据规模是否满足建模需求
→ 完成后报告：附件数据概况（文件数、行列数、缺失值比例、数据类型）
→ ⏸️ 暂停，等用户确认后进入下一步

步骤 1/5：整合方案（验证选型 + 前置合同）
→ 读取设计文档（docs/superpowers/specs/），获取 mmstep2 已选定的模型方案（本步骤只验证不推翻，发现问题报告用户由用户决定）
→ 读取 references/roles/建模手/references/建模设计理论.md，用以下框架验证选型合理性：
  - 三原则：充分性（模型覆盖所有约束）、必要性（最简方案）、可沟通性（非专业人员能理解）
  - 复杂度控制：每个子问题最多 2 个模型、最多 3 层嵌套、最多 5 个手动调参
  - 证据层次：定量验证 > 理论推导 > 专家判断 > 假设推演
→ 参考 assets/ 目录下的算法资源库（7 类 60+ 算法），确认所选算法是否合理（如有明显更优替代方案，报告用户决定是否更换）：
  - 优化问题 → assets/01-优化算法说明.md
  - 预测问题 → assets/02-预测类算法说明.md
  - 评价问题 → assets/03-评价类算法说明.md
  - 图论/网络 → assets/04-图论与网络分析算法说明.md
  - 统计/数据处理 → assets/05-统计分析与数据处理算法说明.md
  - 综合/仿真 → assets/06-综合类算法说明.md
  - 机器学习 → assets/07-机器学习算法说明.md
→ 读取 references/roles/建模手/references/常见模式.md，确认模式匹配是否合理
→ 读取 references/roles/建模手/references/前置合同.md，将结果组织成结构化合同：
  - 子问题-模型映射表（每个子问题对应哪个模型、承担什么证据角色）
  - 方案风险和备选方案
  - 交付规格（给编程手的公式/算法、给论文手的依据/术语）
→ 完成后报告：前置合同概要（含选型验证结论，如有问题需更换模型会明确说明）
→ ⏸️ 暂停，等用户确认后进入下一步

步骤 2/5：搜索文献
→ 读取 tools/paper_search/SKILL.md
→ 为已选模型搜索学术论文，找理论支撑
→ 搜索策略：经典文献（按引用量排序，min_citations=50）+ 最新研究（年份≥2020）+ 数学方法论（field_filter="mathematics"）
→ 完成后报告：找到的参考文献列表（标题、作者、年份、来源、与选型的关联）
→ ⏸️ 暂停，等用户确认后进入下一步

步骤 3/5：执行建模分析
→ 读取 references/roles/建模手/references/工作流程.md，按五步流程执行：
  - 问题理解：确认问题背景、类型、数据、约束（mmstep1 + 步骤 0 已读取，此处确认无遗漏）
  - 模型选择：深化公式推导，给出完整的数学表达式、参数含义、取值范围
  - 算法设计：详细的求解步骤、输入输出格式、边界条件处理
  - 术语规范：建立中英文术语对照表、符号定义
→ 产出：
  - 题目分析报告.md（按工作流程.md 的文档模板格式）
  - 术语表格.md
→ 推荐模型说明图方案（2-3 种），说明每种图的内容和用途：
  - 模型架构图：展示模型整体结构和各模块关系
  - 算法流程图：展示求解算法的步骤和逻辑
  - 几何关系图：展示空间关系、约束条件（如有几何/空间类问题）
  - 其他有助于理解模型的概念示意图
→ 概念图的实际绘制将在 mmstep5 代码实现阶段统一完成（此时模型已确定，不会因后续调整而作废）
→ 完成后报告：各问题的模型和公式概要、推荐的模型说明图方案
→ ⏸️ 暂停，等用户确认后进入下一步

步骤 4/5：缺失输入处理
→ 检查建模分析过程中是否有缺失项（数据不足、模型不适用、问题类型不明确等）
→ 按以下规则处理：
  - 数据不充足 → 标注数据局限性，说明对结果的影响范围
  - 模型不适用 → 记录已排除的模型及原因，给出备选方案
  - 问题类型不明确 → 使用决策树缩小范围，给出最可能的分类及理由
  - 不确定性 → 设计鲁棒性检验方案（参数扰动 ±10%，观察结果敏感性）
  - 需要用户决策 → 整理问题列表，一次性向用户确认
→ 将缺失项和处理方案写入题目分析报告.md
→ 完成后报告：缺失项清单及处理方式
→ ⏸️ 暂停，等用户确认后进入下一步

步骤 5/5：质检交付
→ 读取 references/roles/建模手/references/质检清单.md
→ 逐项检查：
  - A. 前置合同检查：核心结论、子问题拆解、模型映射、防冗余、备选方案
  - B. 模型选择检查：理论依据、数据适配性、复杂度、创新性、三原则
  - C. 算法设计检查：求解步骤、边界条件、评价指标、参数设置
  - D. 交付文档检查：分析文档完整、术语表完整、编程手/论文手交接项清晰
  - E. 风险检查：局限性、数据敏感性、失败预案
→ 完成后报告：质检结果（通过/有问题需修复）
→ ⏸️ 暂停，等用户确认后进入下一步
