# 一次性发送版（全流程）

## 提示词

```
我要做一道数学建模题。请按以下流程执行，每个阶段完成后暂停等用户确认。

⚠️ 不要自动搜索或猜测文件名。必须直接问用户，等用户回答后再操作。
⚠️ 必须严格按以下顺序逐步执行，不得跳步、合并或自行调整顺序。每完成一步后才能进入下一步。每步完成后报告执行结果。

### 第一步：读取文件
- 一次性问用户以下三个文件的信息（不要分三次问）：
  1. 题目文件名（不限格式：PDF/Word/图片等）
  2. 格式要求文件名（不限格式：PDF/Word/图片等）
  3. 附件数据文件名（Excel/CSV），如果没有附件数据用户会说"没有"
- 根据文件扩展名自动选择工具读取（PDF→pdf skill，Word→docx skill，图片→必须派opus子代理识别）
- PDF/Word 中如含示意图、图表等图像内容，必须派opus子代理识别图像部分，prompt 中要求：先判断图片类型（系统拓扑图/流程图/几何示意图/数据图表/公式约束/参数表/地图空间/图例编码）、系统性观察各区域不猜测、提取所有文字数字公式、描述拓扑连接关系、提取图例颜色编码、提取参数值单位约束、有子图则逐个识别，输出按"图片类型→文字→拓扑→参数→图例"分类整理
- 如果有附件数据，使用 xlsx skill 读取（注意中文编码）
- 读取完成后向用户确认题目结构和数据情况

### 第二步：题目分析（使用 brainstorming skill）
- 逐题分析，每题探索 2-3 种建模思路
- 说明优劣和推荐方案
- 每题分析完等用户确认方向
- 全部确认后产出设计文档到 docs/superpowers/specs/ 目录

### 第三步：制定计划（使用 writing-plans skill）
- 读取 brainstorming 阶段产出的设计文档（docs/superpowers/specs/），作为计划的输入
- 基于设计文档中的方案选择，拆分具体任务：
- 拆分可执行子任务（每个 2-10 分钟）
- 规划代码文件结构
- 设计论文大纲
- 标注并行任务和 TDD 范围
- 产出计划文档到 docs/superpowers/plans/ 目录，等用户确认

### 第四步：建模分析（使用 math-modeling skill 建模手角色）
- 主代理没有多模态能力，遇到题目中的图像内容必须派子代理 model: "opus" 识别，其他文本任务主代理处理
- 读取前置产出（如有读取不到的或是缺失的文件，一定要及时反馈！）：设计文档（docs/superpowers/specs/）、计划文档（docs/superpowers/plans/）
- 读取 references/roles/建模手/SKILL.md — 角色入口
- 按以下顺序执行：
  1. 附件检查：检查数据文件夹，用 xlsx/pdf skill 读取附件数据，确认数据充足性
  2. 验证选型 + 前置合同：获取 mmstep2 已选模型（只验证不推翻，发现问题报告用户决定），用建模设计理论验证（三原则、复杂度控制），参考 assets/ 算法资源库（7类60+算法）确认替代方案，组织成前置合同格式（子问题映射、风险、交付规格）
  3. 搜索文献：用 paper_search skill 为已选模型找理论支撑
  4. 执行建模分析：读取工作流程.md，推导公式、设计算法、建立术语表，推荐模型说明图方案（2-3 种），确认后按 nature-figure 设计规则绘图（design-theory.md + common-patterns.md）并做 QA 检查
  5. 缺失输入处理：检查并处理数据不足、模型不适用等问题，用占位符标注缺失项
  6. 质检交付：读取质检清单.md，逐项检查
- 产出：题目分析报告.md（项目根目录）、术语表格.md（项目根目录）、模型说明图（figures/）
- 完成后等用户确认

### 第五步：代码实现（使用 math-modeling skill 编程手角色）
- 主代理没有多模态能力，遇到题目中的图像内容必须派子代理 model: "opus" 识别，其他文本任务主代理处理
- 读取 references/roles/编程手/SKILL.md — 角色入口
- 按以下顺序渐进式读取并执行：
  1. 环境检查：确认 Python 版本和依赖库（numpy, pandas, matplotlib, scipy, openpyxl）
  2. references/roles/编程手/references/工作流程.md → 按流程逐步执行：读取数据 → 编写求解代码 → 运行 → 输出结果表格，参考 assets/ 算法资源库获取代码实现提示
  3. 结果验证：数值合理性检查（NaN/Inf/负数/数量级）、边界条件、与预期对比
  4. 三类可视化规划：每个问题需要原始数据图(EDA)、中间产物图(模型过程)、最终结果图(核心结论)，每类至少 1-2 张，每问题 3-5 张
  5. references/roles/编程手/references/常见模式.md → 参考图表代码模板，确认图表方案
  6. references/roles/编程手/references/可视化规范.md → 获取项目级可视化配置
  7. 通过 Skill 工具调用 nature-figure skill，按五点合同流程绘图（核心结论→证据链→原型选择→后端确认→导出规格），应用 design-theory.md 设计规则和 common-patterns.md 布局模式，中文标签使用 SimHei 或 Microsoft YaHei，SVG + PNG 双格式，按 qa-contract.md 做提交前 16 项检查
  8. references/roles/编程手/references/可视化面板模板.html → 基于模板生成 HTML 图表导航面板，整合所有图表
  9. 生成 README.md（项目简介、目录结构、运行说明）
  10. references/roles/编程手/references/质检清单.md → 交付前逐项检查：代码可运行、结果正确、三类图表完整、图表规范、文件完整
- 读取题目分析报告.md、术语表格.md、计划文档（如有读取不到的或是缺失的文件，一定要及时反馈！）
- 问用户编程语言，然后按问题分文件编写求解代码，最终结果保留 4-6 位小数，图表标签保留 2-3 位小数
- 问用户哪些核心算法需要 TDD，对这些函数使用 TDD
- 运行代码，产出结果表格.csv（results/ 目录）
- 图表保存到 figures/ 目录，代码保存到 code/ 目录
- 在实现公式、运行代码、优化求解、对比验证四个节点检查模型合理性，发现问题暂停告知。小问题直接修正并更新分析报告，大问题等用户确认后调整
- 完成后告知所有文件路径

### 第六步：论文撰写（使用 math-modeling skill 论文手角色）
- 主代理没有多模态能力，遇到题目中的图像内容必须派子代理 model: "opus" 识别，其他文本任务主代理处理
- 读取 references/roles/论文手/SKILL.md — 角色入口
- 按以下顺序渐进式读取并执行：
  1. references/roles/论文手/references/工作流程.md → 构建论证框架：一句话论证模板 + Claim-Evidence 映射表，缺失项用 `[待补充：...]` 占位
  2. references/roles/论文手/references/章节模板.md → 按模板组织各章节结构（摘要六要素、引言五要素、结果证据阶梯）
  3. references/roles/论文手/references/写作规范.md → 禁止第一人称、段落式自然陈述、去 AI 味（8 种模式）
  4. references/roles/论文手/references/进阶技巧.md + 自审框架.md → 段落流检查、动词校准、四轮自审
  5. 问用户论文格式偏好：Word（.docx，国赛常用）或 LaTeX（.pdf，美赛常用）
  6. 确认论文模板：Word 用 默认论文模板.md + docx skill，LaTeX 用章节模板
  7. 读取前置产出（分析报告、术语表、结果表格、图表），如有缺失及时反馈
  8. 撰写论文初稿（正文约 18 页不含附录，附录必须包含完整代码和详细数据，总页数 25-30 页以上），每张图表配 100+ 字分析，写作过程中参照自审框架自检
  9. references/roles/论文手/references/英文化工作流.md（仅美赛）→ 中文论文转化为规范英文
  10. 通过 Skill 工具调用 nature-reviewer skill，按 7 步工作流审查：构建稿件事实库→按 5 维度评估（原创性、科学重要性、跨学科读者兴趣、技术可靠性、可读性）→生成 3 份审稿报告（侧重点不同）→跨审稿综合→QA 检查，遵守 role-boundaries.md 约束（不虚构审稿人身份、不转为作者回应），保存到 paper/审稿报告.md，根据审查意见修改论文
  11. 通过 Skill 工具调用 nature-figure skill，检查图表（SCI/Nature 标准），修正不达标项
  12. Claim-Evidence 映射终检：确认每个论点都有证据支撑
  13. 通过 Skill 工具调用 stop-slop skill，去 AI 味 + 润色全文
  14. 格式检查与最终输出：自审框架第四轮（纸张、字体、图表格式），产出最终论文到 paper/ 目录

### 第七步：打包支撑材料
- 收集支撑材料：源程序（code/）、外部查阅数据、中间结果（results/）、图表（figures/）、分析文档
- 检查合规性：无参赛者身份/学校/赛区信息，无承诺书和编号页
- 检查完整性：核对论文附录引用，源程序可运行
- 将支撑材料文件列表写入论文附录（如无支撑材料则注明"本论文没有支撑材料"）
- 压缩为 ZIP 或 RAR（不超过 20MB），产出支撑材料.zip（项目根目录）

### 可选：并行执行
如果子问题相互独立（>= 2 个），在代码和论文阶段使用
subagent-driven-development 并行处理，加快速度。
```

## 与分步版的区别

| | 分步版 | 一次性版 |
|--|--------|---------|
| 可控性 | 高，每步可调整 | 低，发完就自动跑 |
| 灵活性 | 可根据中间结果调整后续步骤 | 中途调整需要手动干预 |
| 适合场景 | 首次使用、复杂题目 | 熟悉流程后、简单题目 |
| 推荐度 | 推荐 | 了解流程后使用 |
