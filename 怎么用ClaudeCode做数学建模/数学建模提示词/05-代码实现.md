进入代码实现阶段。使用 math-modeling skill 的编程手角色。

⚠️ 主代理没有多模态能力，不能直接读取或识别任何图片。遇到题目中的示意图、图表、手写公式等图像内容，必须派发子代理并指定 model: "opus" 处理。

前置读取：
- 读取编程手的角色文档：math-modeling skill 目录下的 references/roles/编程手/SKILL.md
  - 注意：SKILL.md 中的路径是相对于 math-modeling skill 目录的，以下步骤中的路径是完整路径
- 读取建模分析阶段产出的：题目分析报告.md、术语表格.md（如有读取不到的或是缺失的文件，一定要及时反馈！）
- 读取计划文档（docs/superpowers/plans/ 下的 .md 文件），参考其中的代码文件结构规划和 TDD 标记

执行策略（在开始编码前确认）：

核心算法验证（按需 TDD）：
- 如果之前配置了 TDD 范围，对指定的核心算法函数使用 TDD
- 如果配置为"不需要 TDD"，则跳过，直接编写代码
- 其余脚本代码不做 TDD

并行执行：
- 如果有 >= 2 个相互独立的子问题（问题之间没有数据依赖），问用户是否使用 subagent-driven-development 并行求解
- 如果用户同意，为每个问题派发独立的子 agent，每个子 agent 完成后进行规格审查 + 质量审查
- 如果子问题不独立或只有 1 个问题，顺序执行即可

模型问题处理：
在编码和运行过程中持续检查模型合理性，发现问题立即暂停并告诉用户。

触发节点：
1. 实现公式时：维度/量纲不一致、除零风险、符号冲突、推导跳步
2. 运行代码后：结果出现 NaN/Inf/负数、不符合物理意义、数量级明显错误
3. 优化求解时：目标函数不收敛、约束无法同时满足、陷入局部最优
4. 对比验证时：和附件数据偏差 > 10%、多问题结果矛盾

处理方式：
- 问题明确（公式抄错、符号搞混）→ 直接修正，更新题目分析报告.md，继续
- 问题复杂（模型选型有误、需换方案）→ 暂停，说明问题和判断依据，等用户确认

⚠️ 必须严格按以下顺序逐步执行，不得跳步、合并或自行调整顺序。每完成一步后必须暂停等用户确认，确认后才能进入下一步。

步骤 0/8：环境检查
→ 确认编程语言（由用户在 mmsetup 阶段配置）
→ 检查运行环境：Python 版本、必要的库是否已安装（numpy, pandas, matplotlib, scipy, openpyxl 等）
→ 如果环境有问题，帮助用户安装依赖，不要因为环境问题切换语言
→ 完成后报告：语言版本、已安装的关键库版本
→ ⏸️ 暂停，等用户确认后进入下一步

步骤 1/8：编写求解代码
→ 读取 references/roles/编程手/references/工作流程.md
→ 按流程逐步执行：读取数据 → 编写求解代码 → 运行 → 输出结果表格
→ 参考 assets/ 目录下的算法资源库获取代码实现提示（如适用）
→ 实现公式时如需参考题目中的示意图或图表，必须派发子代理并指定 model: "opus" 识别，prompt 中要求：先判断图片类型（系统拓扑图/流程图/几何示意图/数据图表/公式约束/参数表/地图空间/图例编码）、系统性观察各区域不猜测、提取所有文字数字公式、描述拓扑连接关系、提取图例颜色编码、提取参数值单位约束、有子图则逐个识别，输出按"图片类型→文字→拓扑→参数→图例"分类整理
→ 数值精度要求：最终结果保留 4-6 位小数（和题目数据精度匹配），中间计算使用 Python 默认精度，图表标签保留 2-3 位小数
→ 禁止：简化代码、因报错降低问题难度、因报错切换算法
→ 完成后报告：代码文件名和运行结果概要
→ ⏸️ 暂停，等用户确认后进入下一步

步骤 2/8：结果验证
→ 验证运行结果的合理性：
  - 数值检查：是否有 NaN/Inf/负数、数量级是否合理、是否符合物理意义
  - 边界条件：极值情况是否处理正确、约束是否满足
  - 和预期对比：与题目给出的参考数据（如有）对比偏差
→ 如果发现模型问题，按"模型问题处理"规则处理
→ 完成后报告：验证结论（通过/有问题需修复）
→ ⏸️ 暂停，等用户确认后进入下一步

步骤 3/8：三类可视化规划
→ 按编程手工作流程，每个问题需要三类图表：
  - 原始数据图（EDA）：数据分布、相关性热力图、箱线图等，用于展示数据特征
  - 中间产物图（模型过程）：收敛曲线、模型对比、残差图等，用于展示建模过程
  - 最终结果图（核心结论）：对比柱状图、预测曲线+置信区间、雷达图等，用于展示核心结论
→ 每类至少 1-2 张图，每个问题共 3-5 张图
→ 参考 assets/ 目录下的算法说明（每个算法有"可视化建议"），选择适合的图表类型
→ 向用户推荐图表方案（每类推荐 2-3 种），说明每种方案的适用场景
→ 等用户确认最终的图表数量和内容后，再进入下一步
→ ⏸️ 暂停，等用户确认后进入下一步

步骤 4/8：参考图表模板
→ 读取 references/roles/编程手/references/常见模式.md
→ 参考图表代码模板（折线图、柱状图、热力图、散点图、箱线图、三维曲面等）
→ 为步骤 3 确认的图表方案选择对应的代码模板
→ 完成后报告：各图表对应的模板和实现方案
→ ⏸️ 暂停，等用户确认后进入下一步

步骤 5/8：可视化规范 + 绘图
→ 读取 references/roles/编程手/references/可视化规范.md，获取项目级配置
→ 通过 Skill 工具调用 nature-figure skill（使用 Skill 工具，不要跳过这一步）
→ 按 nature-figure 的五点合同流程绘图：
  - 读取 nature-figure 的 static/core/contract.md，建立五点图表合同：
    1. 核心结论：每张图的一句话主张（带动词的句子）
    2. 证据链：每个面板映射到主张
    3. 原型选择：从以下 4 种中选择——
       · 定量网格（数字比较为主）
       · 示意主导复合（需要理解工作流/机制）
       · 图像板+量化（显微镜/成像主导）
       · 非对称混合模态（综合示意+光栅+热力图+定量图）
    4. 后端：选定 Python，所有绘图在同一后端完成
    5. 导出：SVG + PNG 双格式
  - 读取 nature-figure 的 references/design-theory.md，应用设计规则：
    · 字体：Arial，密集图表 5-7pt，大面板标签 32-54pt
    · 坐标轴：关闭右侧和顶部脊柱，无网格线，图例无边框
    · 调色板：使用语义映射（蓝=提出方法，绿=正面变体，红=基线）
    · 面板标签：小写粗体 a, b, c
  - 读取 nature-figure 的 references/common-patterns.md，选择合适的布局模式
  - 中文标签使用 SimHei 或 Microsoft YaHei 字体，确保不出现乱码
  - 保存为 SVG（可编辑矢量）和 PNG（高清位图，dpi=300）双格式
→ 读取 nature-figure 的 references/qa-contract.md，做提交前 16 项检查，不达标的修正
→ 如果需要视觉检查图表质量（标签重叠、颜色不清晰、数据标注错误），派发子代理并指定 model: "opus" 识别后修正
→ 完成后报告：生成的图表文件列表
→ ⏸️ 暂停，等用户确认后进入下一步
